# Contexto del Proyecto
Actúa como un desarrollador Senior experto en React, TypeScript, TailwindCSS y Firebase. Estás trabajando en "Veterinaria Leo", un sistema de gestión veterinaria. Tienes acceso al código base actualizado.

# Objetivo General
Necesito implementar refactorizaciones críticas enfocadas en la persistencia real de datos en Firebase, la conexión de módulos mediante el uso estricto de parámetros, y la corrección de flujos en la gestión de turnos y usuarios. Es **obligatorio** que todos los datos interactúen con Firestore.

Por favor, analiza los módulos correspondientes y sus servicios (`ParametrosModule`, `PetsModule`, `AppointmentsModule`, `UsersModule`, y la carpeta `services/`) e implementa los siguientes requerimientos paso a paso:

## 1. Bug Crítico: Persistencia en el Módulo de Parámetros
- **Problema actual:** Los datos que agrego en la pestaña de "Parámetros" (Especies, Razas, Vacunas, Eventos Clínicos, etc.) aparecen en la interfaz, pero al salir de la pestaña o recargar, se eliminan.
- **Solución requerida:** Analiza `ParametrosModule.tsx` y `parametrosService.ts`. El problema es que el estado local no se está guardando o recuperando correctamente desde Firebase Firestore. Asegúrate de implementar los métodos `add`, `update`, `delete` y `fetch` en Firestore para toda la configuración de parámetros y que el componente se suscriba a estos cambios (usando `useEffect` para cargar la data inicial al montar el componente).

## 2. Inyección Global de Parámetros (Especies, Razas, Vacunas, etc.)
- **Lógica:** Todo parámetro guardado exitosamente en Firebase (punto 1) debe ser la única fuente de la verdad para el resto del sistema.
- **Implementación:** En componentes como `PetsModule` (Registrar Mascota) o en el historial/vacunación, **elimina los campos de texto abierto**. Sustitúyelos por componentes `Select` o `Combobox` que extraigan sus opciones directamente de la base de datos de Parámetros. Por ejemplo: si selecciono registrar vacuna, el combo debe mostrar las vacunas creadas en parámetros.

## 3. Refactorización de Turnos (Combobox y Selección Múltiple)
- **Motivo de Consulta:** En `AppointmentsModule` (Registrar turnos), elimina el campo de texto libre para "Motivo de la consulta". Reemplázalo por un `Combobox/Select` múltiple que extraiga las opciones desde los "Eventos Clínicos" configurados en Parámetros.
- **Selección Múltiple:** El sistema debe permitir seleccionar **1 o más motivos** (ej: "Consulta Médica" y "Vacunación" en un mismo turno). Ajusta la interfaz y el `turnoService.ts` para que soporte un array de motivos.

## 4. Gestión de Estados del Turno
- **Lógica de Estado:** Cuando un turno ya está registrado y confirmado, la tarjeta o modal de detalle del turno debe mostrar botones de acción lógicos para cambiar su estado.
- **Acciones:** Agrega botones para **"Cancelar turno"** (con confirmación) y **"Turno finalizado"** (cuando ya se atendió). Estos botones deben actualizar el estado del turno directamente en Firebase.

## 5. Refactorización Integral del Registro de Usuarios
- **Problema actual:** El registro funciona a medias pero presenta un error crítico con el mapeo de roles ("veterinario" vs lo que espera el sistema), bloqueando los accesos.
- **Campos Requeridos:** Al crear un **Usuario del Sistema** (no un cliente), el formulario debe solicitar estrictamente: 
  - Nombre
  - Apellido
  - Teléfono
  - Correo electrónico (Mail)
  - Sexo (Debe ser un Select con: Masculino, Femenino, Prefiero no decirlo, Otro)
  - Domicilio
  - Contraseña
  - Perfil/Rol
- **Solución Lógica y de Roles:** Asegúrate de que el proceso de guardado en `userService.ts` o `usuarioService.ts` escriba en Firestore exactamente el mismo `enum` o `string` de ROL que el sistema (ej. `AuthContext` o el Guard de rutas) utiliza para validar los permisos. Revisa la capitalización (ej. si espera "VETERINARIO" o "veterinario" o un ID específico) para que el login posterior sea 100% funcional. Además, asegúrate de que el usuario se cree en Firebase Auth.
